# UniMap 修复实施计划

日期：2026-06-01 | 基于：`docs/PROJECT_REVIEW_2026-06-01.md` | 分支：develop

---

## 总体原则

1. **不破坏现有功能** — 每个修复必须有对应的验证步骤，确保主链路（查询/截图/篡改检测/调度器）不受影响
2. **不破坏依赖链** — 后端修改不走"大重构"路线，保持现有接口签名和调用关系
3. **不破坏程序逻辑链** — 前端修改优先采用增量式事件委托，不动核心查询/结果渲染/WebSocket 逻辑
4. **每阶段可独立合入** — 阶段间无强依赖，任一阶段完成后 `go test -race ./...` 必须全绿

## Phase 3：后端稳定性修复（1 天）

> 目标：修复优雅停机中 scheduler 任务失控的问题。**本阶段风险最高，需充分测试。**

### 3.1 P1 — Scheduler 任务 context 响应 Stop 信号

- **涉及文件**：`internal/scheduler/scheduler.go`
- **改动内容**：
  1. `Scheduler` 结构体增加 `ctx context.Context` 和 `cancel context.CancelFunc`
  2. `NewScheduler()` 初始化 `ctx, cancel = context.WithCancel(context.Background())`
  3. `executeTask()` 中 `context.Background()` → `s.ctx`（第 852 行）
  4. `Stop()` 中，在 `close(s.stopCh)` 后增加 `s.cancel()`
- **技术方案**：

```go
type Scheduler struct {
    // ... 现有字段 ...
    ctx    context.Context
    cancel context.CancelFunc
}

func NewScheduler(store Store, opts ...Option) *Scheduler {
    ctx, cancel := context.WithCancel(context.Background())
    s := &Scheduler{
        // ... 现有初始化 ...
        ctx:    ctx,
        cancel: cancel,
    }
    return s
}

func (s *Scheduler) executeTask(task *ScheduledTask, handler TaskHandler, timeoutSec int, maxRetries int) {
    // ...
    for attempt := 0; attempt <= maxRetries; attempt++ {
        // 检查 scheduler 是否已停止
        select {
        case <-s.ctx.Done():
            record.Status = "cancelled"
            record.Error = "scheduler shutting down"
            // 记录并返回
            return
        default:
        }
        // 原有 logic：
        ctx, cancel := context.WithTimeout(s.ctx, time.Duration(timeoutSec)*time.Second)
        // ...
    }
}

func (s *Scheduler) Stop() {
    s.mu.Lock()
    if s.stopped {
        s.mu.Unlock()
        return
    }
    s.stopping = true
    s.stopped = true
    close(s.stopCh)
    s.mu.Unlock()

    // 取消所有进行中的任务
    s.cancel()

    // 等待 cron 和通知协程
    stopCtx := s.cron.Stop()
    <-stopCtx.Done()
    s.notifyWg.Wait()
}
```

- **依赖链分析**：
  - 关键：`executeTask` 是内部方法，调用方只有 `RunTaskNow` 和 cron callback — 不受影响
  - 关键：`TaskHandler.Execute(ctx, payload)` 中的 `ctx` 现在是 scheduler ctx 的子 context。如果 handler 内部使用了 `context.Background()` 传递给子调用，需要检查 — 当前各 Runner 的 handler 从 `executeTask` 的 ctx 派生，链路正确
  - ICP DB、screenshot router 等资源不在 `executeTask` 中直接访问，在 server.go 的 `Shutdown()` 中顺序是 `httpServer.Shutdown` → `scheduler.Stop()` → `icpDB.Close()`。引入 `s.cancel()` 后，任务在 `icpDB.Close()` 前已收到取消信号并退出，消除竞争窗口
- **验证步骤**：
  1. `go test -race ./internal/scheduler/...` — 必须全绿
  2. 启动服务 → 创建一个耗时 60s 的调度任务 → 立刻 `Ctrl+C` → 确认 scheduler 在 30s 内退出，日志中任务状态为 cancelled
  3. 确认正常停止后进程不残留 goroutine
- **风险**：**中**。改变了 scheduler 的核心生命周期语义。可能有 Runner 不响应 ctx 取消（需要检查每个 Runner 的实现）
- **风险缓解**：`executeTask` 在每次重试前检查 `s.ctx.Done()`，即使 Runner 不响应 ctx，下次重试时也会退出。最长阻塞时间 = 单次任务超时 + 退避间隔
- **回滚**：移除 `s.ctx`/`s.cancel`，`executeTask` 恢复使用 `context.Background()`

```go
// schema.go — 重构后
package database

import (
    "embed"
    "fmt"
    
    "github.com/golang-migrate/migrate/v4"
    _ "github.com/golang-migrate/migrate/v4/database/sqlite3"
    "github.com/golang-migrate/migrate/v4/source/iofs"
)

//go:embed migrations/*.sql
var migrationsFS embed.FS

type Database struct {
    db       *sql.DB
    migrate  *migrate.Migrate
}

func (d *Database) InitSchema() error {
    source, err := iofs.New(migrationsFS, "migrations")
    if err != nil {
        return fmt.Errorf("migration source: %w", err)
    }
    m, err := migrate.NewWithSourceInstance("iofs", source, "sqlite3://"+d.path)
    if err != nil {
        return fmt.Errorf("migrate init: %w", err)
    }
    d.migrate = m
    if err := m.Up(); err != nil && err != migrate.ErrNoChange {
        return fmt.Errorf("migrate up: %w", err)
    }
    return nil
}

func (d *Database) MigrateDown() error {
    if d.migrate == nil {
        return fmt.Errorf("migrate not initialized")
    }
    return d.migrate.Down()
}

func (d *Database) MigrationVersion() (uint, bool, error) {
    if d.migrate == nil {
        return 0, false, fmt.Errorf("migrate not initialized")
    }
    version, dirty, err := d.migrate.Version()
    return version, dirty, err
}
```

- **依赖链分析**：
  - `InitSchema()` 调用方：`web/server.go:330` — 签名不变，无需修改

  - **关键**：现有数据库文件不受影响。`migrate.Up()` 使用 `schema_migrations` 表追踪版本，首次运行创建该表并执行已有 migration

  3. 创建 `2_test.up.sql`（新增一个测试表）→ 启动服务 → 确认新 migration 执行

- **风险**：中。涉及数据库初始化核心路径，需要充分测试新旧数据库兼容性

## 不纳入本次修复的问题

以下问题经评估后决定延后处理，原因如下：

| 问题 | 延后原因 |
|------|---------|
| `config.yaml` 凭证外部化 | 涉及全量环境变量改造，需运维配合，另立项 |
| Kubernetes 部署清单 | 需先完成 Docker 资源限制和 Migration 后才能设计 K8s 配置 |
| 配置热更新并发读写 | 实际触发概率极低（配置极少热更新），优先级低 |
| 前端全局状态重构 | 需要引入状态管理库或模块化方案，单独立项 |
| Docker 镜像漏洞扫描 | 需评估 Trivy/Docker Scout 选型，不影响功能 |
| Nginx 生产化部署 | 需配合运维确定部署拓扑后才能设计 |
| BridgeTokenRotateRunner 功能补齐 | 当前 Extension 模式下令牌轮换通过 API 手动触发，自动轮换需产品确认需求 |
