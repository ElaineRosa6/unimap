[思考步骤 1] 需求解构与边界界定

本次变更拆为四条主线：认证安全补足、前端视觉统一、跨平台兼容、CLI 查询与监控完善。截图能力不进入 CLI 新增范围；已有 CLI 截图子命令先保留兼容，再标记废弃，避免脚本用户突然断链。

需求明确边界非目标
补足认证与安全修改当前管理员密码、认证状态页、API Token/CLI Token 兼容、审计与限流第一阶段不引入完整多租户 RBAC 数据库
前端统一美化按钮、表格、tabs、toast、modal、scheduler/monitor 等页面统一视觉不改 API 字段、不重写页面业务 JS
macOS/Linux 支持路径、Chrome 探测、信号、构建脚本、运行资产打包不承诺 GUI 跨平台无 CGO 依赖
CLI 查询与监控query、monitor reachability、monitor port-scan、认证配置、插件连接选择CLI 不新增截图业务能力
业务线路理顺认证、Web、CLI、Chrome 扩展桥接、定时任务互不抢占鉴权边界不一次性删除旧入口

[思考步骤 2] 变更域划分与域间依赖

依赖顺序：配置与存储层 -> 服务层 -> Web/API 层 -> CLI/前端/Chrome 扩展调用方 -> 文档与发布脚本。认证域会影响 Web、CLI 和插件桥接鉴权，应优先建立适配层；CLI 依赖 Web API 合约稳定；前端美化不得改变接口字段。

  -> internal/auth + web/middleware_auth.go + web/login_handlers.go
  -> web/router.go API
  -> web/templates + web/static/css/style.css

internal/service/query_app_service.go
  -> Web 查询 API
  -> CLI query

internal/service/monitor_app_service.go + monitor_port_scan_service.go

internal/screenshot + web/screenshot_bridge_handlers.go + tools/extension-screenshot
  -> Chrome 扩展桥接

[思考步骤 4] 兼容与过渡方案设计

认证先不替换现有 session、Admin Token、API Key，而是新增 AuthFacade 防腐层统一透传。密码修改先支持当前单管理员配置模型，后续再平滑迁移到用户存储。CLI 先扩展 API 客户端能力和 monitor 子命令，再隐藏 screenshot 子命令。前端先引入设计令牌和按钮语义类，再逐页替换。

层级过渡方案回滚方式
数据/配置层只新增字段，不改旧字段；密码存储继续兼容 web.auth.password_hash忽略新增字段即可
服务层新增认证服务接口，旧校验逻辑透传到新接口中间件改回旧函数调用
接口层新增改密、安全状态、CLI Token 支持；旧登录和 Admin Token 路径保留下线新增路由，不影响旧路由
前端/调用方新增账号安全页和 CLI token 参数；页面 DOM id 保持稳定隐藏入口或恢复旧模板

Chrome 扩展、Web、CLI 连接鉴权设计

这一点必须单独处理。当前项目已经有 Web 登录态/Admin Token/API Key，也有截图桥接路径 /api/screenshot/bridge/ 及 Bridge token 语义。后续不能把它们混成一套 token，否则会出现扩展越权、CLI 误连插件、或普通用户能操作桥接任务的问题。

鉴权边界

调用方访问对象推荐凭据边界要求
浏览器 Web UI管理页面与普通 APIsession cookie + CSRF；可兼容 Admin Token只代表管理员操作，不直接代表 Chrome 扩展身份
CLIWeb APICLI/API Token 或 Admin Token，来自参数、环境变量、配置文件CLI 默认不连接插件；仅在用户选择时检查 bridge 状态
Chrome 扩展Bridge APIPairing token / Bridge bearer token / callback signature独立于 Web session；必须有 TTL、nonce、防重放、可轮换
Web 服务内部BridgeService / ScreenshotRouter内部对象引用，不走外部 token内部调用不暴露给 CLI 直接执行截图

连接模式

模式行为失败策略
plugin_mode=offCLI 不检查、不连接 Chrome 扩展；仅执行 API 查询/监控不受扩展状态影响
plugin_mode=autoCLI 查询 Web API 的 bridge health/status，只展示可用性或启用浏览器采集选项扩展不可用时降级为 API 查询/监控
plugin_mode=requiredCLI 执行前必须确认 Web 上的 BridgeService 可用且扩展已配对不可用时直接失败，返回明确错误码

必须新增的安全约束

  • Bridge API 不应仅因 Web session 存在而放行扩展任务获取和回调。
  • CLI Token 不应复用 Bridge Token；CLI 只向 Web API 证明用户身份，Bridge Token 只向 Web 证明扩展身份。
  • 配对接口必须保留 loopback/origin 检查、短期 token、nonce 和签名校验。
  • Bridge token rotate 必须只允许管理员或内部调度触发，且记录审计日志。
  • CLI 的 plugin_mode=auto 只能做能力发现，不能隐式触发截图任务。
  • Web 页面调用 Bridge 状态接口时，应返回脱敏状态，不返回完整 token。

安全执行切片计划

Slice 0：基线冻结与验收口径

盘点现有接口、配置字段、CLI 参数和页面按钮类，补充 smoke test 清单。只新增测试和文档，不改业务。回滚方式：删除新增测试文档即可。

Slice 1：认证防腐层

新增统一 AuthFacade/AuthService 概念，内部继续调用现有 session、Admin Token、API Key。Web 中间件保持旧入口可用，只把判断逻辑透传到新层。默认关闭新策略开关，保证老配置启动行为不变。

Slice 2：Bridge 鉴权边界加固

在改登录和 CLI 前，先确认 /api/screenshot/bridge/* 的认证边界独立存在。将 Bridge token、pairing token、callback signature 与普通 Web 管理 token 的职责写入接口契约。旧桥接接口先保留路径和字段，只增强校验与审计。

Slice 3：修改密码最小闭环

新增“修改当前管理员密码”服务接口：校验旧密码、bcrypt 新密码、写回配置或安全存储。先不引入多用户。接口层增加 CSRF、限流、审计日志。空配置时沿用现有默认值策略，但页面必须提示需要设置正式密码。

Slice 4：认证管理页面

新增账号安全页面入口，包含修改密码、当前认证方式状态、API Key 只读列表或跳转入口、Chrome 扩展配对状态。旧登录页和退出接口不改路径。若新页面异常，导航入口可隐藏，登录主链路仍可用。

Slice 5：API Key 与 CLI 鉴权适配

CLI API-first 客户端增加 token 读取优先级：命令参数、环境变量、配置文件。请求统一附带 Authorization 或 X-Admin-Token。服务端继续接受旧 header。此切片解决启用 Web Auth 后 CLI 无法调用 API 的风险。

Slice 6：CLI 查询功能收敛

保留 legacy direct-engine 模式；把 API-first query 作为推荐路径。输出格式、分页、错误展示保持兼容。增加连接检测命令，用于判断 Web API、认证、插件桥接是否可用。

Slice 7：CLI 监控功能补齐

新增 monitor reachability 与 monitor port-scan 子命令，对接现有 /api/url/reachability 和 /api/url/port-scan。参数与 Web 页面保持一致：urls、ports、concurrency、timeout、output。失败时返回非零退出码并输出结构化错误。

Slice 8：CLI 插件连接选择

新增 --plugin-mode=off|auto|required。默认 off 或 auto，避免无插件时阻塞查询。required 模式下先检查 bridge health/status，再执行依赖插件的能力。CLI 不提供截图执行入口，只提供连接选择和状态提示。

Slice 9：CLI 截图子命令兼容退场

现有 screenshot-batch 子命令先标记 deprecated，保留一到两个版本；文档说明 CLI 不再新增截图能力。最终删除前必须确认没有脚本依赖。回滚方式：恢复子命令注册即可。

Slice 10：前端设计令牌与按钮体系

先统一 CSS 变量、按钮尺寸、状态色、hover/focus/disabled 样式。不要改接口和页面脚本。重点消除 .btn、.btn-sm、.btn-primary、scheduler 覆盖规则互相冲突的问题。

Slice 11：定时任务与监控页面美化

逐页替换突兀按钮：创建、保存、删除、运行、启停、筛选、导入等使用统一语义类。表格、tabs、summary cards、toast、modal 保持现有 DOM id，避免 JS 断联。

Slice 12：跨平台路径与 Chrome 探测统一

抽出平台能力层：Chrome 查找、打开文件、用户数据目录、临时目录、信号处理。当前 web/cdp_handlers.go 与 internal/screenshot/manager.go 存在重复 Chrome 探测逻辑，应先新增共享适配器，再让旧函数透传。

Slice 13：构建与发布矩阵

补齐 Windows、macOS、Linux 的 Web/CLI 构建验证；GUI 单独标记需要 CGO/Fyne 环境。构建产物必须包含 web/templates、web/static、configs 示例。发布脚本先新增非交互模式，再保留交互模式。

数据与配置演进

本轮不直接改旧字段。新增字段必须有默认值：auth.password_policy、auth.password_changed_at、cli.api_base、cli.plugin_mode、bridge.pairing_required 等。若后续引入 users.json，流程必须是：先加新存储 -> 登录双读 -> 修改密码双写 -> 迁移旧配置 -> 稳定后停止写旧字段 -> 最后删除旧字段。

回滚策略

  • 认证回滚：关闭新 AuthFacade 策略开关，保留旧 session/Admin Token/API Key 判断。
  • Bridge 回滚：恢复旧桥接鉴权路径，但保留 token 轮换审计日志不影响主线。
  • CLI 回滚：恢复旧子命令注册和旧请求客户端，不影响 Web。
  • 前端回滚：恢复 CSS 和模板，不影响服务端接口。
  • 跨平台回滚：旧平台函数仍作为 adapter 后备实现，不直接删除。

[检查清单]

  • 循环依赖：不存在。CLI 只调用 Web API，不反向依赖 CLI；扩展只连接 Bridge API，不依赖 CLI。
  • 破坏性动作缓冲：认证、CLI 截图、Chrome 探测、Bridge 鉴权均先透传再替换。
  • 回滚代价：每个切片可通过配置开关、隐藏入口或恢复路由注册回滚。
  • 业务主线：登录 -> 查询 -> 监控 -> 定时任务 -> 插件桥接不被中断。
  • 鉴权隔离：Web session、CLI/API Token、Bridge Token 必须分域使用，不能混用。

建议验收顺序

先跑 Web 登录与查询 smoke test，再跑 CLI query/monitor，再跑 Bridge pairing/health/status，再跑 scheduler 页面按钮操作，最后分别在 Windows、macOS、Linux 验证 Web/CLI 启动、路径处理、Chrome 探测和插件连接状态。
